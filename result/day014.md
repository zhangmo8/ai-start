# Day 014 - Week 02 复盘与进度更新

## 本周主题

Python 工程基础

## 我已经理解了什么

这一周真正解决的问题，不是再多学几个 Python 库，而是建立“一个真实脚本为什么不仅要能跑，还要能被理解、被排查、被复现”的工程视角。

我现在已经能把这一周的内容按 3 层串起来：

- `接口表达层`：`typing` 与函数签名，负责把输入输出和调用边界表达清楚
- `运行时故障处理层`：异常处理，负责让错误能被发现、传播、拦截和定位
- `工程骨架层`：`pathlib / json / argparse / logging / venv`，负责入口、资源定位、配置管理、可观测性和环境隔离

我也已经能明确说出这几组知识各自解决什么问题：

- `typing` 解决“接口怎么写清楚”
- 异常处理解决“运行时报错怎么办”
- `pathlib` 解决“路径和资源怎么稳定管理”
- `json` 解决“配置和结构化信息怎么交换”
- `argparse` 解决“脚本怎么被参数化启动”
- `logging` 解决“运行过程怎么留下可追踪线索”
- `venv` 解决“同一份代码如何在更可控的环境里运行”

我现在也已经能从模型工程主线角度理解这周的价值：

- 训练脚本如果没有清楚接口，协作会很难
- 推理脚本如果没有异常边界，失败时只能靠猜
- 实验如果没有配置，很难复现
- 任务如果没有日志，很难排障

所以 Week 02 的价值，不是记住几个标准库名称，而是开始形成“我知道一个真实 Python 工程最小需要哪些骨架”的感觉。

## 这周内容和主线的关系

这一周和后面“自己做小模型、读开源项目、接 agent / MCP、做推理和评测”的关系非常直接。

- 后面读模型代码时，不只是看算法，还要看脚本入口、配置加载、日志输出和错误处理。
- 后面写工具或 agent 时，不只是关心功能，还要关心输入输出边界是否清楚、失败时能不能定位问题。
- 后面做训练和评测时，真正影响效率的常常不是模型公式本身，而是配置是否可复现、路径是否稳定、日志是否足够。

所以这一周补的其实是“工程可持续性”的基础，而不是零散语法点。

## 我还不会什么

当前还没完全吃透的，不再是最表层概念，而是更细一点的工程边界：

1. `typing` 里更复杂的泛型、协议和类型变量，什么时候值得引入，什么时候会过度设计。
2. 稍大项目里 `Logger / Handler / Formatter` 应该怎样组织，才能既统一又不混乱。
3. 配置系统在项目继续变大后，什么时候应该从简单 `json` 升级到更完整的校验和配置管理方案。

## 下周最大风险是什么

下周最大的风险不是“学不会 NumPy”，而是把 `线性代数与 NumPy` 误学成两种偏差中的一种：

1. 只把它当成学校数学公式，停留在抽象符号层，不知道它和真实数组、shape、模型代码有什么关系。
2. 只把它当成 NumPy API 记忆题，会写一点 `reshape` 和 `dot`，但没有建立维度变化和广播的直觉。

如果带着这两种偏差进入 Week 03，就会出现一种很常见的问题：

- 能看懂单行代码
- 但看不懂张量为什么长这个 shape
- 也说不清一次矩阵运算之后维度为什么会变成那样

所以下周最需要补的洞，不是先记很多函数名，而是建立“数据形状如何组织、如何变化、为什么这样变化”的直觉。

## 本周最模糊的 3 个问题

1. Python 工程里，`type hint` 和运行时校验的边界应该怎么划，哪些错误应交给类型表达，哪些错误必须靠运行时检查。
2. 一个稍大一点的项目里，日志配置到底应该集中在入口统一处理，还是允许模块各自扩展，这条边界怎么拿捏。
3. 当实验和脚本越来越多时，简单配置文件、参数解析、结构校验之间应该怎样逐步升级，而不是一上来就过度设计。

## 下周开始前最需要记住的一句话

Week 02 让我知道真实 Python 工程最小需要哪些骨架；Week 03 要开始理解，模型代码里的数据到底是如何被组织、变形和计算的。
